---
title: "Middleware"
description: "Intercept and extend agent behavior at key lifecycle points"
---

Middleware allows you to inject custom logic — logging, tracing, input modification, access control, and more — into an agent's execution pipeline without modifying the agent or model code.

## Overview

AgentScope's middleware system provides **5 interception points** covering the full Agent lifecycle, from the outer reply process down to the raw model API call. Each middleware is optional: you only implement the hooks you need, and the system automatically detects which hooks are active at runtime.

The hooks follow two patterns:

- **Onion Pattern** (4 hooks): `on_reply`, `on_reasoning`, `on_acting`, `on_model_call` — each wraps the next handler, allowing before/after logic and stream observation.
- **Transformer Pattern** (1 hook): `on_system_prompt` — middlewares run sequentially, each transforming the system prompt string and passing it to the next.

The following diagram illustrates how middleware hooks wrap the agent lifecycle:

```
+--------------------------------------------------+
| on_reply                                         |
|  +----------------------------------------------+|
|  | on_reasoning                                 ||
|  |  +------------------------------------------+||
|  |  | on_model_call                            |||
|  |  |   +--------------------------------+     |||
|  |  |   |     Model.__call__()           |     |||
|  |  |   +--------------------------------+     |||
|  |  +------------------------------------------+||
|  |  +------------------------------------------+||
|  |  | on_acting                                |||
|  |  |   +--------------------------------+     |||
|  |  |   |     Tool execution             |     |||
|  |  |   +--------------------------------+     |||
|  |  +------------------------------------------+||
|  +----------------------------------------------+|
|                                                  |
|  on_system_prompt: prompt -> mw1 -> mw2 -> final |
+--------------------------------------------------+
```

## Quick Start

The following example creates a simple middleware that logs before and after each model call:

```python
from agentscope.middleware import MiddlewareBase
from agentscope.agent import Agent

class LoggingMiddleware(MiddlewareBase):
    """Log before and after model calls."""

    async def on_model_call(self, agent, input_kwargs, next_handler):
        print(f"[{agent.name}] Calling model...")
        result = await next_handler()
        print(f"[{agent.name}] Model call completed.")
        return result

agent = Agent(
    name="assistant",
    system_prompt="You are a helpful assistant.",
    model=model,
    toolkit=toolkit,
    middlewares=[LoggingMiddleware()],
)
```

## Hook Reference

### on_reply

Intercepts the **entire reply process**, including all reasoning rounds, tool executions, and the final output message.

```python
async def on_reply(
    self,
    agent: Agent,
    input_kwargs: dict,
    next_handler: Callable[..., AsyncGenerator],
) -> AsyncGenerator:
```

The `input_kwargs` dictionary contains the following keys:

| Key | Type | Description |
|-----|------|-------------|
| `msgs` | `Msg \| list[Msg] \| None` | The input messages triggering the reply |
| `event` | `UserConfirmResultEvent \| ExternalExecutionResultEvent \| None` | Optional event context |

Wrap `next_handler()` and yield each item to observe the reply process. Calling `next_handler()` without arguments forwards the original input unchanged:

```python
class ReplyMiddleware(MiddlewareBase):
    async def on_reply(self, agent, input_kwargs, next_handler):
        print("Reply started")
        async for item in next_handler():
            yield item
        print("Reply finished")
```

You can modify input by passing keyword overrides to `next_handler(...)`:

```python
from agentscope.message import UserMsg

class ModifyInputMiddleware(MiddlewareBase):
    async def on_reply(self, agent, input_kwargs, next_handler):
        original_msg = input_kwargs["msgs"]
        modified_msg = UserMsg(
            name=original_msg.name,
            content="[Prefix] " + original_msg.content,
        )
        async for item in next_handler(
            msgs=modified_msg,
            event=input_kwargs.get("event"),
        ):
            yield item
```

### on_reasoning

Intercepts the **reasoning phase** — the model call that produces text, thinking, and/or tool call decisions within a single ReAct round.

```python
async def on_reasoning(
    self,
    agent: Agent,
    input_kwargs: dict,
    next_handler: Callable[..., AsyncGenerator],
) -> AsyncGenerator:
```

The `input_kwargs` dictionary contains the following keys:

| Key | Type | Description |
|-----|------|-------------|
| `tool_choice` | `ToolChoice \| None` | The tool choice configuration for this reasoning step |

<CodeGroup>
```python Basic usage
class ReasoningMiddleware(MiddlewareBase):
    async def on_reasoning(self, agent, input_kwargs, next_handler):
        print("Reasoning started")
        async for event in next_handler():
            yield event
        print("Reasoning finished")
```

```python Modifying tool_choice
from agentscope.tool import ToolChoice

class ForceNoToolMiddleware(MiddlewareBase):
    """Prevent the model from calling tools."""

    async def on_reasoning(self, agent, input_kwargs, next_handler):
        async for event in next_handler(
            tool_choice=ToolChoice(mode="none"),
        ):
            yield event
```
</CodeGroup>

### on_acting

Intercepts **individual tool call execution**. Called once per tool call within a reasoning round.

```python
async def on_acting(
    self,
    agent: Agent,
    input_kwargs: dict,
    next_handler: Callable[..., AsyncGenerator],
) -> AsyncGenerator:
```

The `input_kwargs` dictionary contains the following keys:

| Key | Type | Description |
|-----|------|-------------|
| `tool_call` | `ToolCallBlock` | The tool call to execute (name, arguments, id) |

<CodeGroup>
```python Basic usage
class ToolLoggingMiddleware(MiddlewareBase):
    async def on_acting(self, agent, input_kwargs, next_handler):
        tool = input_kwargs["tool_call"]
        print(f"Executing tool: {tool.name}")
        async for event in next_handler():
            yield event
        print(f"Tool {tool.name} completed")
```

```python Modifying tool call arguments
from agentscope.message import ToolCallBlock

class SanitizeToolInputMiddleware(MiddlewareBase):
    async def on_acting(self, agent, input_kwargs, next_handler):
        tool_call = input_kwargs["tool_call"]
        sanitized = ToolCallBlock(
            id=tool_call.id,
            name=tool_call.name,
            input=sanitize(tool_call.input),
            state=tool_call.state,
        )
        async for event in next_handler(tool_call=sanitized):
            yield event
```
</CodeGroup>

### on_model_call

Intercepts the **raw model API call** — the lowest-level hook, sitting right above the actual `ChatModelBase.__call__()`.

Unlike the other onion hooks which yield async generators, `on_model_call` **returns** a `ChatResponse` or `AsyncGenerator[ChatResponse, None]`.

```python
async def on_model_call(
    self,
    agent: Agent,
    input_kwargs: dict,
    next_handler: Callable[
        ...,
        Awaitable[ChatResponse | AsyncGenerator[ChatResponse, None]],
    ],
) -> ChatResponse | AsyncGenerator[ChatResponse, None]:
```

The `input_kwargs` dictionary contains the following keys:

| Key | Type | Description |
|-----|------|-------------|
| `messages` | `list[Msg]` | The formatted messages for the model |
| `tools` | `list[dict]` | The tool schemas |
| `tool_choice` | `ToolChoice \| None` | The tool choice configuration |
| `current_model` | `ChatModelBase` | The model instance for this call |

<CodeGroup>
```python Non-streaming usage
class TimingMiddleware(MiddlewareBase):
    async def on_model_call(self, agent, input_kwargs, next_handler):
        import time
        start = time.time()
        result = await next_handler()
        elapsed = time.time() - start
        print(f"Model call took {elapsed:.2f}s")
        return result
```

```python Streaming usage
from agentscope.model import ChatResponse

class StreamLoggingMiddleware(MiddlewareBase):
    async def on_model_call(self, agent, input_kwargs, next_handler):
        result = await next_handler()

        if isinstance(result, ChatResponse):
            return result

        async def wrapped():
            async for chunk in result:
                print(f"Chunk: {chunk.is_last}")
                yield chunk

        return wrapped()
```

```python Modifying model input
from agentscope.message import Msg

class InjectSystemMessageMiddleware(MiddlewareBase):
    async def on_model_call(self, agent, input_kwargs, next_handler):
        messages = [
            Msg(name="system", content="Extra instruction", role="system"),
        ] + input_kwargs["messages"]

        return await next_handler(
            current_model=input_kwargs["current_model"],
            messages=messages,
            tools=input_kwargs["tools"],
            tool_choice=input_kwargs["tool_choice"],
        )
```
</CodeGroup>

### on_system_prompt

Transforms the **system prompt string**. Unlike other hooks, this uses a **transformer/pipeline pattern** — middlewares are applied sequentially, each receiving the output of the previous one.

```python
async def on_system_prompt(
    self,
    agent: Agent,
    current_prompt: str,
) -> str:
```

```python
class AppendContextMiddleware(MiddlewareBase):
    async def on_system_prompt(self, agent, current_prompt):
        return current_prompt + "\n\nCurrent time: 2025-01-01T00:00:00"
```

Multiple `on_system_prompt` middlewares are applied left to right:

```python
agent = Agent(
    ...
    middlewares=[mw1, mw2],  # prompt → mw1 → mw2 → final
)
```

<Note>
`on_system_prompt` may be called **more than once** per reply — once during initial setup and again during each reasoning step.
</Note>

## Execution Order

### Onion Pattern

For onion hooks (`on_reply`, `on_reasoning`, `on_acting`, `on_model_call`), the **first middleware in the list is the outermost layer**:

```python
middlewares = [mw1, mw2]

# Execution order:
# mw1 before → mw2 before → [inner logic] → mw2 after → mw1 after
```

For streaming/yielding hooks, the inner middleware sees each yielded item first:

```
mw1_pre → mw2_pre → mw2_event → mw1_event → ... → mw2_post → mw1_post
```

### Transformer Pattern

For `on_system_prompt`, middlewares run **sequentially left to right**:

```python
middlewares = [mw1, mw2]

# Execution:
# original_prompt → mw1.on_system_prompt() → mw2.on_system_prompt() → final
```

## Combining Multiple Hooks

A single middleware can implement multiple hooks. The system automatically detects which hooks are overridden and routes them appropriately.

```python
class FullObservabilityMiddleware(MiddlewareBase):
    """Observe all lifecycle phases."""

    async def on_reply(self, agent, input_kwargs, next_handler):
        print(f"[reply] start for {agent.name}")
        async for item in next_handler():
            yield item
        print(f"[reply] end for {agent.name}")

    async def on_reasoning(self, agent, input_kwargs, next_handler):
        print(f"[reasoning] start")
        async for event in next_handler():
            yield event
        print(f"[reasoning] end")

    async def on_model_call(self, agent, input_kwargs, next_handler):
        print(f"[model_call] calling {input_kwargs['current_model'].model}")
        result = await next_handler()
        print(f"[model_call] done")
        return result

    async def on_system_prompt(self, agent, current_prompt):
        print(f"[system_prompt] length={len(current_prompt)}")
        return current_prompt

agent = Agent(
    ...
    middlewares=[FullObservabilityMiddleware()],
)
```

The same middleware instance appears in multiple internal hook lists. Execution order of hooks within a single reply follows the agent's lifecycle:

```
on_reply
  └── compress_context() → on_system_prompt (1st call, token counting)
  └── on_reasoning
        └── _prepare_model_input() → on_system_prompt (2nd call, build model input)
        └── on_model_call
  └── on_acting (for each tool call)
```
**Note:** `on_system_prompt` is triggered twice per reasoning round: first
from `compress_context()` during token counting (before `on_reasoning`), and
again from `_reasoning_impl()` when building the actual model input. In a
multi-step ReAct loop, this pattern repeats for each round.

## Built-in: TracingMiddleware

AgentScope provides a built-in `TracingMiddleware` that adds **OpenTelemetry tracing** to the full agent lifecycle — reply, model calls, and tool executions. It implements three hooks (`on_reply`, `on_model_call`, `on_acting`) and produces hierarchical spans following the [OpenTelemetry GenAI Semantic Conventions](https://opentelemetry.io/docs/specs/semconv/gen-ai/).

### Setup

Before using `TracingMiddleware`, call `setup_tracing()` to configure the OTLP exporter endpoint:

```python
from agentscope.middleware import TracingMiddleware, setup_tracing

setup_tracing(endpoint="http://localhost:4318/v1/traces")
```

Then attach `TracingMiddleware` to your agent:

```python
from agentscope.agent import Agent

agent = Agent(
    name="assistant",
    system_prompt="You are a helpful assistant.",
    model=model,
    toolkit=toolkit,
    middlewares=[TracingMiddleware()],
)
```

### Span Hierarchy

When tracing is enabled, each agent reply produces a nested span tree:

```
agent reply (on_reply)
  ├── llm call (on_model_call)
  ├── tool execution (on_acting)   ← one per tool call
  ├── llm call (on_model_call)     ← follow-up reasoning
  └── ...
```

### Captured Attributes

<Tabs>
  <Tab title="Agent Reply Span">
  Captured via `on_reply`:
  - Agent name, session ID, reply ID
  - Input messages and final output message
  - HITL (Human-in-the-Loop) pending tools
  - External execution pending tools
  </Tab>
  <Tab title="Model Call Span">
  Captured via `on_model_call`:
  - Model name, provider, input/output tokens
  - Request/response message content
  - Streaming-aware: wraps async generators to capture the final chunk
  </Tab>
  <Tab title="Tool Execution Span">
  Captured via `on_acting`:
  - Tool name, call ID, input arguments
  - Tool execution result
  </Tab>
</Tabs>

### Zero-Overhead When Disabled

If `setup_tracing()` has not been called, every hook short-circuits directly to `next_handler()` with minimal overhead — no spans are created, no attributes are computed.

<Info>
When the agent receives an `ExternalExecutionResultEvent` (tools executed outside the agent), `TracingMiddleware` creates synthetic spans for each externally-executed tool result, preserving full observability even for tools that run in external systems.
</Info>

## Practical Examples

### Simple Timing Middleware

The following middleware logs all model interactions with timing:

```python
import time
from agentscope.middleware import MiddlewareBase

class TimingMiddleware(MiddlewareBase):
    async def on_model_call(self, agent, input_kwargs, next_handler):
        model_name = input_kwargs["current_model"].model
        start = time.time()

        result = await next_handler()

        elapsed = time.time() - start
        print(f"[timing] {agent.name} → {model_name}: {elapsed:.2f}s")
        return result
```

### Rate Limiting Middleware

The following middleware enforces a delay between model calls:

```python
import asyncio
from agentscope.middleware import MiddlewareBase

class RateLimitMiddleware(MiddlewareBase):
    def __init__(self, min_interval: float = 1.0):
        self._last_call = 0.0
        self._min_interval = min_interval

    async def on_model_call(self, agent, input_kwargs, next_handler):
        import time
        now = time.time()
        wait = self._min_interval - (now - self._last_call)
        if wait > 0:
            await asyncio.sleep(wait)
        self._last_call = time.time()
        return await next_handler()
```

### Dynamic System Prompt Middleware

The following middleware injects real-time context into the system prompt:

```python
from datetime import datetime
from agentscope.middleware import MiddlewareBase

class DynamicContextMiddleware(MiddlewareBase):
    def __init__(self, context_fn):
        self._context_fn = context_fn

    async def on_system_prompt(self, agent, current_prompt):
        context = self._context_fn()
        return f"{current_prompt}\n\n## Current Context\n{context}"

agent = Agent(
    ...
    middlewares=[
        DynamicContextMiddleware(
            lambda: f"Time: {datetime.now().isoformat()}"
        ),
    ],
)
```

### Model Fallback Middleware

The following middleware switches to a fallback model when the primary fails:

```python
from agentscope.middleware import MiddlewareBase

class ModelFallbackMiddleware(MiddlewareBase):
    def __init__(self, fallback_model):
        self._fallback = fallback_model

    async def on_model_call(self, agent, input_kwargs, next_handler):
        try:
            return await next_handler()
        except Exception as e:
            print(f"Primary model failed: {e}, switching to fallback")
            return await next_handler(
                current_model=self._fallback,
                messages=input_kwargs["messages"],
                tools=input_kwargs["tools"],
                tool_choice=input_kwargs["tool_choice"],
            )
```

## API Reference

### MiddlewareBase

```python
class MiddlewareBase:
    def is_implemented(self, hook_name: str) -> bool
    async def on_reply(self, agent, input_kwargs, next_handler) -> AsyncGenerator
    async def on_reasoning(self, agent, input_kwargs, next_handler) -> AsyncGenerator
    async def on_acting(self, agent, input_kwargs, next_handler) -> AsyncGenerator
    async def on_model_call(self, agent, input_kwargs, next_handler) -> ChatResponse | AsyncGenerator
    async def on_system_prompt(self, agent, current_prompt) -> str
```

All hooks raise `RuntimeError` by default. The `is_implemented()` method checks whether a subclass has overridden a hook by comparing the method reference against `MiddlewareBase`'s definition. The agent uses this at construction time to build separate hook lists, avoiding unnecessary calls for unimplemented hooks.

### Agent Integration

Middleware is configured via the `middlewares` parameter on `Agent.__init__`:

```python
Agent(
    name="assistant",
    system_prompt="...",
    model=model,
    toolkit=toolkit,
    middlewares=[mw1, mw2, mw3],  # Applied in order
)
```

The agent internally partitions the middleware list by implemented hooks:

| Internal List | Hooks |
|---------------|-------|
| `_reply_middlewares` | Middlewares implementing `on_reply` |
| `_reasoning_middlewares` | Middlewares implementing `on_reasoning` |
| `_acting_middlewares` | Middlewares implementing `on_acting` |
| `_model_call_middlewares` | Middlewares implementing `on_model_call` |
| `_system_prompt_middlewares` | Middlewares implementing `on_system_prompt` |

<Info>
This partitioning is done **once** at construction time for efficiency.
</Info>
